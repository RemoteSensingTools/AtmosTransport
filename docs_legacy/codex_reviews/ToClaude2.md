**Findings**
1. Critical: the new evolving-mass pilot corrupts its own flux field by aliasing the scaled flux array and the CFL scratch buffer. In X, the caller passes `ws.cfl_x` as `am_eff` at [mass_flux_advection.jl#L1523](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1523), then `_x_pilot_succeeds` reuses that same `ws.cfl_x` as the output buffer for `max_cfl_massflux_x` at [mass_flux_advection.jl#L1492](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1492). The same bug exists for Y at [mass_flux_advection.jl#L1580](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1580) and [mass_flux_advection.jl#L1551](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1551), and Z at [mass_flux_advection.jl#L1654](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1654) and [mass_flux_advection.jl#L1613](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1613). That means the pilot and then the real sweep can end up using a mutated CFL field instead of the intended `am/n_sub`, `bm/n_sub`, `cm/n_sub`.

2. High: the new X pilot does not model the same mass evolution as the real reduced-grid X advection. The pilot mass kernel at [mass_flux_advection.jl#L428](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L428) ignores `cluster_sizes` entirely and updates fine-grid `m` with `am[i]-am[ip]`. But the X CFL check at [mass_flux_advection.jl#L716](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L716) and the real X sweep at [mass_flux_advection.jl#L1108](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/mass_flux_advection.jl#L1108) are reduced-grid aware. So even after fixing the aliasing bug, the pilot can still accept or reject based on an X mass path the runtime never actually follows.

3. High: `prognostic_slopes=true` is still silently ignored in the live lat-lon path. The flag is parsed into `SlopesAdvection` at [configuration.jl#L968](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/configuration.jl#L968), but `_apply_advection_latlon!` explicitly strips any `ws.base` wrapper and always dispatches to `strang_split_massflux!` at [run_helpers.jl#L223](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/run_helpers.jl#L223). So configs like [era5_f64_nofixer_2day.toml#L36](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/era5_f64_nofixer_2day.toml#L36) do not test the advertised prognostic-slope path.

4. Medium: the commit message overstates “TM5-faithful”. The live LL path still clamps `cm` at [physics_phases.jl#L744](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L744) and still zeros `am` on the first/last latitude rows at [physics_phases.jl#L757](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L757). Those are explicit stabilizers, not parity with TM5 forward transport.

5. Medium: the newly committed “nofixer” config is still misleading as a reference case. [era5_f64_nofixer_2day.toml#L27](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/era5_f64_nofixer_2day.toml#L27) sets `dry_ic = true`, [era5_f64_nofixer_2day.toml#L36](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/era5_f64_nofixer_2day.toml#L36) sets `prognostic_slopes = true` even though that path is disabled, and [era5_f64_nofixer_2day.toml#L38](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/era5_f64_nofixer_2day.toml#L38) still has `mass_fixer = true`. It is not a clean TM5-faithful no-fixer baseline.
