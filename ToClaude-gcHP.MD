**Findings**
1. High: the live GEOS “GCHP” path is **not fully GCHP-faithful** because horizontal subcycling is global, not per-level. In [cubed_sphere_fvtp2d_gchp.jl#L917](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/cubed_sphere_fvtp2d_gchp.jl#L917) through [cubed_sphere_fvtp2d_gchp.jl#L923](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/cubed_sphere_fvtp2d_gchp.jl#L923), `gchp_tracer_2d!` computes one scalar `nsplt` from the global max `|CX|,|CY|`. The file itself claims “per-level subcycling,” and upstream GCHP does this per level, not as one global worst-case split.

2. High: the common **dry-basis GEOS path is intentionally not GCHP-truthful**. The active dry branch in [physics_phases.jl#L951](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L951) through [physics_phases.jl#L976](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L976) and [physics_phases.jl#L1090](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1090) through [physics_phases.jl#L1105](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1105) uses direct-cumsum target PE plus column scaling and surface locking, not pure GCHP DryPLE / hybrid PE. The repo even has the pure GCHP-style DryPLE/hybrid machinery available in [vertical_remap.jl#L1332](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/vertical_remap.jl#L1332) and [vertical_remap.jl#L1380](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/vertical_remap.jl#L1380), but that is not what the dry branch is using.

3. High: even the moist branch still has a real GCHP mismatch because advected humidity is disabled. In [physics_phases.jl#L1223](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1223) through [physics_phases.jl#L1228](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1228), the code explicitly disables QV advection as tracer `NQ+1`. Then back-conversion uses met-data `QV_next/QV_curr` in [physics_phases.jl#L1425](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1425) through [physics_phases.jl#L1465](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1465). That is an approximation, not full GCHP parity.

4. Medium: the geometry in the “GCHP-faithful” horizontal transport still uses a documented approximation. [cubed_sphere_fvtp2d_gchp.jl#L57](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/cubed_sphere_fvtp2d_gchp.jl#L57) and [cubed_sphere_fvtp2d_gchp.jl#L224](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/cubed_sphere_fvtp2d_gchp.jl#L224) say `sin_sg` is approximated as `1.0`, with about a 5% corner error. But the public docstring later says “exact sin_sg” in [cubed_sphere_fvtp2d_gchp.jl#L405](/home/cfranken/code/gitHub/AtmosTransportModel/src/Advection/cubed_sphere_fvtp2d_gchp.jl#L405). That claim is wrong.

5. Medium: the live GCHP path is reliable only if the met source already contains `CX/CY/XFX/YFX`. The branch gate in [physics_phases.jl#L1520](/home/cfranken/code/gitHub/AtmosTransportModel/src/Models/physics_phases.jl#L1520) only checks that the buffers exist, not that they were populated with valid data. Binary mode can load embedded Courant and area-flux fields in [geosfp_cs_met_driver.jl#L1488](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/geosfp_cs_met_driver.jl#L1488) through [geosfp_cs_met_driver.jl#L1495](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/geosfp_cs_met_driver.jl#L1495), and the reader supports them in [binary_readers.jl#L442](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/binary_readers.jl#L442) and [binary_readers.jl#L467](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/binary_readers.jl#L467). But the NetCDF-side `load_cx_cy_window!` helper is separate and not wired into `load_all_window!` in [geosfp_cs_met_driver.jl#L1674](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/geosfp_cs_met_driver.jl#L1674), so I would not trust generic NetCDF GCHP mode without checking the actual load path.

6. Medium: many “GCHP” configs are actually defaulting to the repo’s **dry** transport basis, not the more faithful moist one. The config builder maps `dry_correction=true` to `pressure_basis="dry"` by default in [configuration.jl#L328](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/configuration.jl#L328) through [configuration.jl#L331](/home/cfranken/code/gitHub/AtmosTransportModel/src/IO/configuration.jl#L331). For example, [catrine_geosit_c180_gchp_v4_7d.toml#L68](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/catrine_geosit_c180_gchp_v4_7d.toml#L68) through [catrine_geosit_c180_gchp_v4_7d.toml#L75](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/catrine_geosit_c180_gchp_v4_7d.toml#L75) is dry, while [catrine_geosit_c180_gchp_v4_7d_perremap_moist.toml#L68](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/catrine_geosit_c180_gchp_v4_7d_perremap_moist.toml#L68) through [catrine_geosit_c180_gchp_v4_7d_perremap_moist.toml#L77](/home/cfranken/code/gitHub/AtmosTransportModel/config/runs/catrine_geosit_c180_gchp_v4_7d_perremap_moist.toml#L77) is the closer parity target.

**My take**
I would not call the current GEOS path “GCHP-truthful” without qualification. The moist `gchp=true`, `vertical_remap=true`, `per_step_remap=true`, `pressure_basis="moist"` path is the closest one. The dry path is a deliberate variant, not parity.

The biggest remaining GEOS parity gaps are:
- global `nsplt` instead of per-level `ksplt`
- `sin_sg = 1` geometry approximation
- no live QV tracer advection in the moist path
- fragile dependency on embedded `CX/CY/XFX/YFX`
